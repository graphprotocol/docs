> Пожалуйста, обратите внимание, что размещенный сервис начнет работу в первом квартале 2023 года, но он останется доступным для сетей, которые не поддерживаются в децентрализованной сети. Разработчикам рекомендуется [переносить их сабграфы](https://thegraph.com/blog/how-to-migrate-ethereum-subgraph) по мере поддержки большего количества сетей. Каждая сеть будет постепенно закрывать свои эквиваленты размещенных сервисов, чтобы у разработчиков было достаточно времени для переноса сабграфов в децентрализованную сеть. Узнайте больше о запуске размещенного сервиса [здесь](https://thegraph.com/blog/sunsetting-hosted-service).

В этом разделе вы узнаете, как развернуть сабграф в [Размещенном сервисе](https://thegraph.com/hosted-service/).

Если у вас нет учетной записи на размещенном сервисе, вы можете зарегистрироваться с помощью своей учетной записи Github. Как только вы пройдете аутентификацию, вы можете начать создавать сабграфы через пользовательский интерфейс и развертывать их со своего терминала. Размещенный сервис поддерживает ряд evm-совместимых сетей, таких как Polygon, Gnosis Chain, BNB Chain, Optimism и Arbitrum. Полный список смотрите в разделе [Поддерживаемые сети](https://thegraph.com/docs/en/deploying/deploying-a-subgraph-to-hosted/)

## Создать сабграф

Сначала следуйте инструкциям [здесь](/developing/defining-a-subgraph), чтобы установить графический интерфейс. Создайте сабграф, передав в `graph init --product hosted-service`

### Из существующего контракта

Если у вас уже есть смарт-контракт, развернутый в основной сети Ethereum или в одной из тестовых сетей, загрузка нового сабграфа из этого контракта может быть хорошим способом начать работу с размещенным сервисом.

Вы можете использовать эту команду для создания сабграфа, который индексирует все события из существующего контракта. Это попытается извлечь ABI контракта из [Etherscan](https://etherscan.io/).

```sh
graph init \
  --product hosted-service
  --from-contract <CONTRACT_ADDRESS> \
  <GITHUB_USER>/<SUBGRAPH_NAME> [<DIRECTORY>]
```

Кроме того, вы можете использовать следующие необязательные аргументы. Если ABI не может быть извлечен из Etherscan, он возвращается к запросу локального пути к файлу. Если в команде отсутствуют какие-либо необязательные аргументы, она проведет вас через интерактивную форму.

```sh
--network <ETHEREUM_NETWORK> \
--abi <FILE> \
```

`<GITHUB_USER>` в данном случае - это имя вашего пользователя GitHub или организации, `<SUBGRAPH_NAME>` - это имя вашего сабграфа, а `<DIRECTORY>` - необязательное имя каталога, в который graph init поместит пример манифеста сабграфа. `<CONTRACT_ADDRESS>` - это адрес вашего существующего контракта. `<ETHEREUM_NETWORK>` - это название сети Ethereum, в которой действует контракт. `<FILE>` - это локальный путь к файлу ABI контракта. **Оба параметра --network и --abi являются необязательными.**

### Из примера сабграфа

Второй режим, который поддерживает `graph init`, - это создание нового проекта из примера сабграфа. Следующая команда делает это:

```
graph init --from-example --product hosted-service <GITHUB_USER>/<SUBGRAPH_NAME> [<DIRECTORY>]
```

Пример сабграфа основан на контракте Gravity от Dani Grant, который управляет пользовательскими аватарами и выдает события `NewGravatar` или `UpdateGravatar` всякий раз, когда создаются или обновляются аватары. Сабграф обрабатывает эти события, записывая объекты `Gravatar` в хранилище нод Graph и обеспечивая их обновление в соответствии с событиями. Перейдите к [манифесту сабграфа](/developing/creating-a-subgraph#the-subgraph-manifest), чтобы лучше понять, на какие события из ваших смарт-контрактов следует обратить внимание, сопоставления и многое другое.
